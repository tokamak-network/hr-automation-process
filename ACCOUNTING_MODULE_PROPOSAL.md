# 회계 자동화 모듈 제안서 (작업 지시서)

> **Tokamak Network HR Solution에 회계 자동화 모듈을 추가하자는 제안**
>
> 싱가포르 법인의 가결산(YA2026 + YA2027)을 직접 수행한 경험을 바탕으로 작성. 실제 거래 데이터·거래처·금액 등 민감 정보는 이 문서에서 제외함(작업 지시 목적). 필요 시 별도 보안 경로로 제공.

---

## 작성 배경

법인 2년치 가결산을 직접 작업한 결과, 회계연도 1년치 분개 + BS/PL 작성에만 수 시간, 자료 수집에 며칠이 소요됨. 매년 반복되는 이 시간을 자동화로 줄이는 것이 목표. 더 큰 문제는 **자료가 매년 사후적으로 재구성**된다는 점 — 시간이 지나면 거래 내역 추적이 어렵고, 인보이스·계약서·영수증이 흩어져 audit 대응이 힘들다.

**목표: 기존 채용/급여 모듈과 연결되는 회계 모듈을 추가해, 반복되는 가결산 작업을 자동화한다.**

---

## 1. 문제 정의 — 가결산에서 시간을 많이 쓴 작업들

각 항목은 자동화 대상이 되는 실제 병목이다. 괄호 안은 필요 기능.

### 1.1 거래 분류 (최대 병목)
수백 건 거래를 회계 계정으로 분류. 대부분 "거래상대 이름"으로 분류 가능하나, 분류 룰이 사람 기억에만 존재해 매년 반복. 신규 거래상대가 나오면 매번 사람이 판단. 여러 통화·계좌의 거래내역을 각각 받아 통합해야 함.
→ **필요**: 거래상대별 회계 계정 매핑 테이블(학습형), 신규 거래상대 자동 질문 → 1회 답변 후 자동 분류, 매월 자동 분류 + 마감 시 검토.

### 1.2 환율 수집 (단조 반복)
일별 환율을 장기간 수동 수집. 마감일이 휴일이면 직전 영업일 환율로 보정. 매출 인식은 거래일 환율, 마감 외화 잔액 평가는 마감일 환율 — 두 시점 모두 필요.
→ **필요**: 일별 환율 자동 수집(무료 API), KRW 페어 포함, 주말·공휴일 보정 룰 자동 적용.

### 1.3 인보이스/외주 송금 추적 (흩어진 정보 재조립)
매출 인보이스 발행 기록이 없어 입금만 보고 인식하다 AR(수취채권) 누락 위험. 외주 송금은 거래내역만으로 역추적(PO·송장 없음). 계약서 유무에 따라 정확도 차이 큼.
→ **필요**: 인보이스 모듈(발행일·거래상대·금액·통화·입금예정일·실입금일), 벤더/외주 모듈(거래상대 마스터·송금 이력·PO 첨부), 계약서 저장소, 그리고 "이 송금 ↔ 이 계약서/PO" 매핑.

### 1.4 직원 보상 (회계 누적 기록 부재)
급여는 일정하나 회계연도별 합계가 별도 기록되지 않아 거래내역에서 합산해야 함. 기존 Payroll 모듈은 지급을 추적하나 회계 관점 누적은 별도.
→ **필요**: 월별 급여 → 회계연도별 자동 집계, 토큰 인센티브의 환산 시점 환율 보존, 이사보수 별도 추적, 퇴직금 적립(해당 시).

### 1.5 자산 관리 (일괄 처리 한계)
고정자산의 상세 내역이 없어 일괄 처리.
→ **필요**: 자산 등록부(구입일·자산명·단가·내용연수·상각방식), 자산별 NBV 자동 계산, 폐기/매각 이력, 연 1회 감가상각 분개 자동 생성.

### 1.6 BS 균형 디버깅 (추적 어려움)
가결산 중 BS 불일치 디버깅에 시간 소요. 원인은 전기 마감 잔액이 당기 opening balance에 반영 안 됨. Plug 분개가 매년 발생하나 추적 어려움.
→ **필요**: 분개 입력 시 Dr=Cr 자동 검증, 마감 시 BS 균형 자동 확인, Plug 분개 별도 추적, 차기 opening balance 자동 이월.

### 1.7 한국 원천세 검토 (누락 위험 1순위)
한국 거주자에게 USD 송금 시 원천징수 의무 가능성(한-싱 조세조약). 현재 미검토 상태로 audit 리스크.
→ **필요**: 거래상대 마스터에 "세금 거주지" 필드, 한국 거주자 송금 자동 플래그, 원천세 계산 보조, 거주증명서 보관.

### 1.8 보고서 출력 (마지막 단계 자동화 가능)
BS/PL PDF, 한국어 가결산 보고서, 종합 워크북 등 산출물은 버튼 한 번에 자동 생성 가능한 영역.

## 2. 모듈별 제안 (M1~M10)

각 모듈의 목적과 핵심 요구사항만 명시. 구현 시 이 규칙을 따른다.

### Module 1 — Bank Transaction Sync
**목적**: 은행 거래내역 자동 수집 → DB 저장.
**요구**: Public API를 제공하는 은행·핀테크는 자동 동기화. API 미제공 계좌는 CSV 자동화 또는 수동 업로드 폴백. 다통화(SGD/USD/GBP/KRW) 지원. 계좌 간 자금이체 중복 제거.
**연결**: M2(분류), M3(인보이스).

### Module 2 — Transaction Classifier ⭐ 최우선
**목적**: 거래상대 → 회계 계정 매핑을 학습/저장.
**요구**: 거래상대 마스터 테이블. 신규 거래상대 나오면 사용자에게 분류 요청 → 답을 학습해 자동 분류. 이름 매칭 룰(정규식/유사도, 표기 변형 인식). 분류 룰 변경 이력(audit). 계정 체계는 기존 chart of accounts 따름.
**Bonus**: AI가 ref/memo까지 보고 분류 후보 제안.

### Module 3 — Invoice & Vendor
**목적**: 매출 인보이스 발행 / 외주 벤더 관리.
**요구(매출)**: 발행일·거래상대·금액·통화·due date·실입금일. 발행 → 입금 자동 매칭. 마감일 미입금분 AR 자동 인식.
**요구(매입)**: 벤더 마스터(이름·거주지·기본 계정·계약서 첨부), 송금 이력 + PO/Invoice, 정기 송금 자동 인식.

### Module 4 — Contract & Document Repository
**목적**: audit에 필요한 모든 서류를 한 곳에.
**요구**: 카테고리별 보관 — 근로계약서, 이사보수계약서, 외주계약서, 차입/대여계약서, 임대차계약서, 인보이스, 영수증, 자본금 서류, 거주증명서, 보증금 영수증.
**핵심 가치**: audit 질문에 계약서 PDF를 즉시 제시 가능.

### Module 5 — FX Rate Service
**목적**: 일별 환율 자동 수집 → SGD/KRW 환산.
**요구**: 일별 mid rate API(무료·무인증 권장). 통화 페어 USD/SGD, GBP/SGD, KRW/USD·SGD, USDT(=USD), 토큰(거래소 API). 마감일 휴일 보정. 분개 시 거래일 환율 자동 적용.

### Module 6 — Compliance Calendar
**목적**: 반복 신고/마감 일정 자동 알림.
**요구**: 주요 일정 관리 — ECI, Corporate Tax(Form C/C-S), GST, ACRA Annual Return, 4대보험(해당 시), 한국 원천세(해당 시), 차입 만기, 가결산 마감. 각 일정별 D-알림.
**연결**: 텔레그램/이메일/Slack (기존 알림 구조 활용).

### Module 7 — Korean WHT Helper
**목적**: 한국 거주자 송금 시 원천징수 의무 검토 자동화.
**요구**: 거래상대 마스터 "거주지" 필드(Korea/Singapore/Others), 한국 거주자 송금 자동 플래그, 원천세 계산 가이드(한-싱 조세조약), 거주증명서 보관 + 만료 알림, 신고 이력.

### Module 8 — Fixed Asset Register
**목적**: 자산 등록 + 자동 감가상각.
**요구**: 자산 등록(자산명·카테고리·구입일·단가·통화·내용연수·상각방식), 정액/정률 자동 계산, NBV 추적, 폐기/매각 이력, 마감 시 감가상각 분개 자동 생성.

### Module 9 — Financial Statements Generator
**목적**: 가결산 → BS/PL PDF 자동 출력.
**요구**: 거래+분개 → GL → PL/BS 자동 빌드. 영문 BS/PL(ACRA 양식), 한국어 가결산 보고서, 3년 추세 비교, 종합 워크북 xlsx.
**연결**: weasyprint/reportlab. 기존 보고서 생성 코드 참고 가능.

### Module 10 — Audit Trail
**목적**: 모든 회계 변동 이력을 audit 가능하게.
**요구**: 분개 입력/수정/삭제 이력, 분류 룰 변경 이력, 자산 폐기/매각 승인 흐름, BS 균형 자동 검증(Dr=Cr), 마감 후 잠금(이전 연도 수정 불가, audit 시 unlock).

## 3. 기존 모듈과의 연결

- **Payroll(기존)**: 급여·인센티브·소득세 적립 → Salary 회계 분개로 자동 연결. 송금 트랜잭션 → 자동 거래 등록. 세금 적립 → Tax Accrual 부채로 BS 반영.
- **Hiring(기존)**: 채용 → 입사 → Salary 마스터 자동 생성. 후보자 정보 → Vendor 마스터 후보(외주 계약 시).
- **사용자/권한(기존)**: 역할 → 회계 권한 매핑(예: 관리자만 회계연도 closing 가능).

---

## 4. 우선순위 (Phase)

**Phase 1 — Quick Win (1–2개월)**: M2(분류) → M3(인보이스/벤더) → M4(계약서 저장소). 이 셋만으로 가결산 시간 50%+ 절감.

**Phase 2 — Mid-term (3–6개월)**: M1(은행 동기화), M5(환율), M6(컴플라이언스 캘린더), M7(한국 WHT).

**Phase 3 — Long-term (6–12개월)**: M8(고정자산), M9(재무제표 생성), M10(감사 추적).

## 5. 단 한 가지를 꼽으면 — Module 2 (Transaction Classifier)

가결산 작업의 대부분이 "이 거래는 어떤 회계 계정인가?"를 결정하는 시간. 그 결정은 거의 전부 거래상대 이름으로 가능하며, 한 번 학습시키면 다음 해부터 자동. **가장 큰 시간 절감 효과 → 최우선 구현 대상.**

---

## 안전선 (구현 시 준수)

1. **판단은 사람, 실행/기록은 시스템.** 신규 거래 분류는 사람이 승인 → 시스템이 학습·기록. 자동으로 확정하지 않는다.
2. **실데이터·거래처·금액·직원 정보는 public repo에 커밋 금지.** 이 모듈은 회사의 재무 구조를 다루므로, 매뉴얼·지시서·코드에 실데이터를 남기지 않는다.
3. **감사 가능성.** 모든 회계 변동은 이력을 남긴다(누가·언제·무엇을). "주장하지 말고 기록으로 증명".
4. **기존 모듈과 동일한 로컬/클라우드 분리·모드 표시 원칙**을 따른다.
