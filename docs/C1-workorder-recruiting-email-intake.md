# C-1 작업지시서 — 채용 지원 메일 자동 감지·등록

> [!summary] 목표 / Goal
> `hr@tokamak.network`로 들어오는 Track B 지원 메일을 감지해, repo·지갑·이름을 추출하고, **검토 게이트를 거쳐** Candidates에 등록한다. 수기 입력을 대체(병행 유지). 다음 단계인 온보딩 연결의 토대.

> [!danger] 절대 안전선 / Hard lines
> 1. **Gmail 읽기 전용.** 메일 발송·삭제·수정 없음. 자동 회신 절대 없음.
> 2. **자동 등록 없음.** 감지→검토 게이트→사람 확인 후 등록.
> 3. **데이터·키 repo 커밋 금지** (public). DB는 Supabase, 백엔드 경유. RLS 기본 거부 유지.
> 4. 지원 단계 정보(이름·이메일·repo·지갑)는 저민감. **온보딩 단계 개인정보는 별도 강한 설계**(이 문서 범위 밖).

---

## §0. 배경 — Cowork 검증에서 나온 핵심 교훈

실데이터(최근 30일 hr@) 검증 결과 / lessons from live test:
1. **지원 1건(Vijay)**, 협업·뉴스레터 등 ~200건은 필터로 전량 제외됨 → 2중 필터 유효.
2. **repo와 지갑이 다른 메일에 온다** — repo는 1차 제출(5/25), 지갑은 후속 메일(6/11). → **메일 단위가 아니라 "지원자(발신자) 단위"로 누적**해야 한다.
3. 0x hex 중 **컨트랙트 주소·트랜잭션 해시는 지갑이 아님** → 지갑 검증 필요.
4. 유입이 적다(월 1건 수준) → **가볍게**. 과도한 정교화 금지, 핵심 흐름만 견고히.

---

## §1. candidates 테이블 확장

현재 `candidates`에 **지갑주소 칸이 없음**(공고 필수 항목인데). 추가:

| 추가 컬럼 | 타입 | 용도 |
|---|---|---|
| wallet_address | TEXT | ERC-20 지갑주소 (없으면 NULL='지갑 대기') |
| source | TEXT | 유입 경로 (`email_auto` / `manual` / `monitor` 등) |
| source_email_id | TEXT | 감지된 Gmail 메시지/스레드 ID(중복·추적용) |
| detected_at | TEXT | 자동 감지 시각 |

- 기존 `email`(발신자 이메일)을 **지원자 식별 키**로 사용(같은 사람의 여러 메일 누적).

---

## §2. Gmail 읽기 연동 (읽기 전용)

- 개인 Gmail(`hr@tokamak.network`), **읽기 전용 권한만**(이미 그렇게 연결됨).
- **검색으로 범위 한정**(전체 메일함 스캔 금지): 최근 N일(기본 14일) + 지원 신호 키워드(`github.com` OR `0x` OR `repository`/`submission`/`Track B` 등). 협업·뉴스레터까지 다 읽지 않게.
- Gmail 접근 자격증명은 백엔드 `.env`(커밋 금지).

---

## §3. 지원 감지 — 2중 필터 + 지원자 단위 누적

### 감지 규칙
1. **1차(내용 필터)**: 메일에 **GitHub repo 링크**(`github.com/{owner}/{repo}`) 또는 **ERC-20 지갑주소**가 있으면 후보 신호.
2. **지갑 검증**: `0x` + 40 hex 이면서, **컨트랙트 주소·트랜잭션 해시(잘린 64자 등)는 제외.** (Cowork가 구분한 로직 반영)
3. **지원자 단위 누적**: 발신자 이메일 기준으로 묶는다.
   - repo만 있는 1차 메일 → 후보 생성, `wallet_address=NULL`(지갑 대기).
   - 후속 메일에서 지갑이 오면 → **같은 발신자 후보에 지갑 채움**(새 후보 만들지 말 것).
4. **제외**: repo·지갑 둘 다 없는 메일(협업·문의·뉴스레터)은 후보 대상 아님. 건수만 집계.

### 분류 결과
- **지원 감지**: repo 있음(+지갑 있으면 완비). 검토 게이트로.
- **확인 필요**: 애매한 건(repo 불명확, 지갑 형식 의심 등). 자동 등록 금지.

---

## §4. 검토 게이트 (자동 등록 없음)

- 감지된 후보는 바로 candidates에 넣지 말고 **"감지됨(검토 대기)" 목록**에 올린다.
- 운영자가 화면에서 확인 후 **"등록 승인"** → candidates에 `status='submitted'`, `source='email_auto'`로 등록.
- 중복 방지: `source_email_id`/발신자 이메일로 이미 등록·감지된 건 재생성 안 함.
- **수기 입력 병행 유지**: 자동 감지 못 한 예외는 기존 수기 등록 경로로.

---

## §5. 스케줄 — 주 2회 자동 + 수동 트리거

- **주 2회 자동 스캔**(예: 월·목). 경비 스케줄러(`expense_scheduler.py`) 패턴 재사용.
- **수동 트리거 엔드포인트**(`POST /api/candidates/scan-inbox`) + 화면 버튼("새 지원 확인").
- 스캔은 **감지·목록화·알림까지만**. 등록은 사람이 검토 후. (자동 등록·회신 없음)
- 새 감지 시 알림(텔레그램 등 기존 채널 재사용).

---

## §6. 기존 Candidates·평가 연결

- 등록된 후보는 기존 흐름 그대로 → `analyze`(AI 평가) → review → reward.
- 즉 C-1은 **입구만** 자동화. 평가·리뷰·보상은 기존 시스템 활용.
- 화면: Candidates 목록에 `source`(email_auto/manual) 표시, "감지됨(검토 대기)" 섹션 추가.

---

## §7. 완료 기준 (DoD)

- [ ] candidates에 wallet_address·source·source_email_id·detected_at 추가.
- [ ] Gmail 읽기 연동(읽기 전용), 검색 범위 한정.
- [ ] 2중 필터 + 지갑 검증(컨트랙트/해시 제외).
- [ ] 지원자(발신자) 단위 누적 — repo 선등록, 후속 지갑 채움.
- [ ] 검토 게이트(감지→승인→등록), 자동 등록·회신 없음.
- [ ] 주 2회 자동 + 수동 트리거 버튼.
- [ ] 등록 후보가 기존 analyze로 이어짐. 수기 입력 병행 유지.

---

## §8. 검증 시나리오 (구현 후)

1. Vijay 케이스 재현: 5/25 repo 메일 → 후보 생성(지갑 대기), 6/11 지갑 메일 → **같은 후보에 지갑 채워지는가**(새 후보 안 생기는가).
2. 협업·뉴스레터 메일 → 후보로 안 올라오는가(제외 건수만).
3. 0x 컨트랙트 주소/트랜잭션 해시가 든 메일 → 지갑으로 오인하지 않는가.
4. 수동 트리거 버튼 → 즉시 스캔되는가. 주 2회 자동도 동작하는가.
5. 검토 게이트: 감지만으론 candidates에 안 들어가고, 승인 시 들어가는가.
6. Gmail 권한이 읽기 전용인지, 발송/삭제 호출이 0건인지.
