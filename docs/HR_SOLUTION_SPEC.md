# Tokamak HR Solution — 확정 스펙

## 컨셉
A(기존 Hiring 확장) + B(미니멀 UI) + D(자동화 워크플로우) 하이브리드

## 회사 구조
- ~10명, 독립계약자 형태
- 인건비: USDT 월 1회 (매월 마지막 영업일)
- 인센티브: TOKAMAK 토큰 분기 1회
- 원천징수 없음, 근로소득세 기준 시뮬레이션 → 과세분 토큰 적립금 회사 보유

## 급여 워크플로우
1. **D-7**: 예상 비용 산출 → 텔레그램 알림 (Kevin 보고용 자료)
2. **D-1**: 전날 종가 기준 급여 내역 확정 → 텔레그램 알림
3. **세금 계산**: 근로소득세 시뮬레이션 → 적립금 산출 → 실 지급액 확정
4. **지급**: Kevin 계정(메타마스크) → Jaden 계정 입금 → Jaden이 각 팀원에게 지급
5. **기록**: 지갑 주소 기반 트랜잭션 자동 조회 + 히스토리 저장

## 가격 기준
- USDT/KRW: 전날 종가
- TOKAMAK/KRW: 전날 업비트 종가
- 이유: 토큰 변동성 고려, 당일이 아닌 전날 종가로 빠른 업무 처리

## 소득세 계산

### 기본 원칙
- 근로소득세율표 기준 시뮬레이션
- 보수적 과세 → 연말에 대부분 환급이 되도록 운영
- 과세분에 해당하는 토큰(USDT/TOKAMAK) = 세액 KRW ÷ 전날 시세 → 적립금으로 회사 보유

### 월별 과세 (USDT 급여)
- 연간 USDT 예상액 기준 세율 적용
- 월별 세액 산출 → 적립

### 분기별 과세 (TOKAMAK 인센티브)
- USDT 연간 예상액 + 해당 분기까지 누적 인센티브 합산 기준 세율 적용 (B방식)
- 분기마다 누적 합산으로 세율 재조정 → 점점 정확해짐
- 분기 세액 산출 → 적립

### 연말토큰정산 (12/31 이후)
- 실제 연간 총소득(USDT + TOKAMAK) 확정 → 정확한 세율 재계산
- 기적립 세액과 비교
- 기적립 > 실제 세액 → 환급 (대부분 이 케이스가 되도록 보수적 과세)
- 기적립 < 실제 세액 → 추가 과세 가능 (완전 불가는 아님, 발생 시 적용)

### 퇴사정산
- 퇴사 시점이 해당 팀원의 연말정산 역할
- 근무기간 실제 총소득(USDT + TOKAMAK) 기준 세액 재계산
- 기적립 > 실제 세액 → 환급
- 기적립 < 실제 세액 → 추가 공제
- 추가 공제 > 마지막 급여인 경우 → 회사가 흡수 (팀원에게 청구하지 않음)

## 기술 스택
- Frontend: Next.js (기존 Hiring 구조 확장)
- Backend: FastAPI (기존 Hiring 구조 확장)
- DB: SQLite → PostgreSQL 마이그레이션 권장
- 외부 API: 업비트 API (TOKAMAK 종가), 환율 API (USDT/KRW), Etherscan/블록 탐색기 API (트랜잭션 조회)
- 알림: Telegram Bot API
- 배포: 웹 애플리케이션
