# 경비 정산 에이전트 + HR 3계층 아키텍처

> [!summary] 한 줄
> 기존 `hr-automation-process` 백엔드는 **진실의 원천(데이터·도구)**으로 두고, 그 위에 ① 일을 하는 에이전트, ② 얇은 운영자 콘솔, ③ 얇은 팀 포털을 나란히 올린다. 첫 에이전트는 **경비 정산**으로 패턴을 증명한다.

---

## 1. 3계층 아키텍처

### 공통 토대 — 백엔드 + 데이터 모델 (기존 자산)
- 기존 FastAPI + SQLite를 유지하되 역할을 **"앱의 두뇌"에서 "도구·데이터 계층"으로** 강등.
- 모든 계산·정산·기록은 여기서 일어나고, 위 세 계층은 이걸 읽거나 호출만 한다.
- 가능하면 핵심 기능을 API 엔드포인트로 정리하고, 에이전트가 호출하기 쉽게 MCP 서버로 감싼다(선택).

### Layer 1 — 에이전트 (일을 하는 주체) ★신규
- 트리거를 받아 스스로 다단계 작업을 수행하고, 초안까지 만들어 사람에게 올린다.
- 이번 문서의 경비 정산 에이전트가 1호.

### Layer 2 — 운영자 콘솔 (Member 2 전용, 얇게)
- 역할: 에이전트가 올린 **승인·예외 검토**. 직접 처리 기능을 추가하지 않는다.
- 에이전트 도입 후 기능이 **줄어드는** 게 정상. 여기에 로직을 붙이려는 충동이 제자리걸음의 원인.

### Layer 3 — 팀 셀프서비스 포털 (팀원용, 얇게) ★신규
- 역할: 팀원이 **본인** 데이터를 조회. 초기엔 payslip 조회 + 이번달 예상세금 2개만.
- 업무 로직 없음. 이미 계산된 결과를 사용자별로 표시만 한다.

> [!important] UI / 백엔드 경계 규칙 (한 줄)
> **UI는 보여주고(display) 호출(trigger)할 뿐, 계산하거나 결정하지 않는다.**
> UI가 "세금을 어떻게 계산하는가" 같은 로직을 품으려 하면 → 백엔드/에이전트 몫.
> 프론트에 로직을 넣고 있다고 느껴지면 그게 제자리걸음 신호.

UI가 하는 일은 4가지뿐: (1) 사용자별로 스코핑된 데이터 표시 (2) 입력 수집 (3) 엔드포인트·에이전트 호출 (4) 상태 표시.

## 2. 경비 정산 에이전트 — 직무기술서

> [!note] 책임 (한 문장)
> 팀원이 제출한 경비 증빙을 수집·파싱·정규화하여 **건별 결정용 요약과 보류 목록**을 만들고, Member 2의 건별 지급 판단 후 기록·통지까지 처리한다.

> [!important] 전제: 지급여부는 규칙이 아니라 재량 판단
> 사내 경비정책은 카테고리·한도 규칙이 아니라 "업무 관련 지출인가"에 대한 **Member 2의 재량 판단**이다. 따라서 에이전트는 **자동 지급하지 않는다.** 에이전트의 목표는 결정의 *개수*를 줄이는 게 아니라, 건당 판단에 드는 *수고*를 줄이는 것(첨부 열람·환산·중복확인 → 한 줄 훑어보기).

### 트리거
- 1차(이벤트): 팀원이 증빙 제출(메일 / Slack / Drive 폴더).
- 2차(시간): 매월 정해진 정산일(예: 매월 말일) 자동 마감 실행.

### 입력
- 증빙 파일: 영수증·명세 (PDF / 이미지 / HWP·HWPX 가능).
- 제출자, 제출일, 신고 금액·통화·사유(있으면).
- 참조: 사내 경비정책 문서(카테고리·한도·필수증빙 규칙).

### 도구 (Tools)
- **문서 파서**: 영수증/명세 → 구조화 텍스트. 이미지=OCR, 한글 HWP/PDF=kordoc류 MCP.
- **채널 리더**: 메일/Slack/Drive에서 제출물 수집.
- **환율 조회**: 정산 기본 단위 = **USD**. 비-USD 통화(EUR/KRW/SGD/crypto 등)를 USD로 환산. **2개 기준 병기** — ① 추산=각 건 결제일 종가, ② 확정=지급일 D-1 종가(지급일 확정 후).
- **과거 승인 이력 참조**: Member 2의 이전 지급/보류 결정 (맥락 제공·후기 추천 학습용, 선택).
- **백엔드 기록 API**: 정산 결과 DB 기록.
- **알림**: 텔레그램(기존 급여 워크플로우와 동일 채널).

### 실행 단계 (에이전트 자율, 80%)
1. 신규 제출물 수집 (이미 처리한 건 제외 — *상태/기억* 참조).
2. 각 증빙 파싱 → 금액·일자·항목·통화 추출.
3. 카테고리 분류 (travel / equipment / software / meals / etc).
4. 비-USD → **USD** 환산(**추산**): 각 건의 **결제일(영수증상 결제 완료일) 종가** 적용. 결제일이 증빙에 없으면 임의 추정 금지 → 보류. (이미 USD인 건은 환산 불요)
5. 결정용 정규화: 항목당 "누가·얼마·무엇을·왜·증빙상태"를 한 줄로 압축 + 판단 보조 맥락 부착(예: 동일인 최근 유사 지출 횟수).
6. 보류 표시(형식 문제만): 증빙 누락·판독 불가, 신고액≠증빙액, 중복 의심, 환율 모호. *업무 관련성은 판단하지 않음.*
7. 건별 결정용 목록 + 보류 목록 생성. (지급 확정분은 이후 인별 합계로 집계)

### 판단 게이트 (Member 2, 20%)
- 지급여부는 Member 2의 재량 판단이므로 **건별로 본다**(예외만 보는 게 아님). 대신 에이전트의 결정용 요약 덕분에 건당 판단이 "훑어보기"로 단축.
- 각 건: **지급 / 보류 / 추가증빙 요청** 중 선택.
- 지급 결정 시: 지급일을 잡고, **확정 환율 = 지급일 D-1 종가**로 환산값을 채워 그 값으로 지급. (수집 시점엔 추산값만, 확정값은 이때 채움)
- (선택·후기) 과거 결정이 쌓이면 에이전트가 "지급 추천 / 확인 필요"로 **사전 분류만** 제안 — 자동 지급 없음, 최종 결정권은 항상 Member 2.
- 지급 확정 시 → 에이전트가 DB 기록 + 텔레그램 통지(기존 급여 흐름과 합류).

### 예외 처리 규칙
- 증빙 누락/판독 불가 → 보류 목록으로, 추가증빙 요청.
- 금액 불일치(신고액 ≠ 증빙액) → 표시 후 보류.
- 중복 의심(동일 제출자·금액·일자) → 표시 후 보류.
- 결제일 불명 → 추산 환율 산정 불가이므로 **임의 추정 금지**, 보류(결제일 명시 증빙 요청).
- 업무 관련성 모호 → **임의 판단 금지**, Member 2 결정 대기.

### 상태 / 기억 (cross-run)
- 이번 달 이미 처리한 제출물 식별(중복 처리 방지).
- 인별 과거 이력(중복·이상 패턴 탐지).
- 반복 항목(구독료 등) 인지.

### 출력
- 인별 결정용 표(제출자 · 금액·원통화 · **추산 USD(결제일 종가)** · **확정 USD(지급일 D-1 종가, 지급 확정 후)** · vendor/항목 · 사유 · 증빙상태).
  - 추산·확정 두 칼럼을 병기해 환차를 한눈에. 수집 시점엔 확정 칼럼 = "지급일 미정".
- 보류 목록(사유 포함).
- (지급 확정분) 인별 합계.
- 승인 후: DB 기록 + 텔레그램 통지.

### 성공 기준
- 건별 지급 판단이 "첨부 열어 영수증 읽고 환산하고 중복 확인"에서 **한 줄 훑어보기**로 단축(결정 수는 그대로, 건당 노력 급감).
- 월 마감 소요시간 단축.
- 지급 결정 이력이 깨끗한 데이터로 누적 → 팀 포털·재무보고 재사용 + (후기) 추천 학습 근거.

## 3. 팀 셀프서비스 포털 — 초기 범위

> [!warning] 범위를 좁게 고정한다
> 1차 출시는 아래 2개만. 운영자 콘솔에서 겪은 무한 확장 함정을 반복하지 않는다.

### 포함 (2개)
1. **payslip 조회** — 에이전트/급여 모듈이 산출한 본인 명세 표시(읽기 전용).
2. **이번달 예상세금** — 세금 엔진(기존 MVP)이 산출한 본인 추정치 표시(읽기 전용).

### 명시적 비범위 (out of scope, 1차)
- 경비 제출 UI, 근태, 퇴사 정산, 다운로드 외 편집 기능, 관리자 기능.

### 보안 / 데이터 격리 요구 (1순위)
- payslip·세금은 민감 개인·금융정보. 팀 대상 공개 즉시 운영 부담 발생.
- 기존 역할 기반 다중사용자 토대 위에 **"각자 본인 데이터만 본다"** 격리를 확실히 건다.
- 모든 조회는 read-only. 인증·HTTPS·최소 노출 원칙.
- 비개발자 장기 운영 부담을 줄이려면 기능을 좁게 유지하는 게 보안상으로도 유리.

## 4. 구축 순서

1. **경비 에이전트로 패턴 증명** + 깨끗한 정산 데이터 축적.
2. 같은 데이터 위에 **팀 포털(2기능)**을 얇게 붙인다(병행 또는 직후).
3. 직무기술서를 **템플릿화** → 급여·채용, 이후 재무보고(ETH 트레저리)·계약 disclosure 검토로 복제.
4. 복제가 곧 "HR → 기타영역 확장" = 회사 지원팀 구성.
